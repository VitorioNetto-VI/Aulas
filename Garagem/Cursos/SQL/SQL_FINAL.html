Window Functions, CTEs, Particionamento e Segurança em Produção

Avançado Conclusivo

Chegamos ao ápice do curso. Aqui, exploraremos ferramentas analíticas que substituem lógicas complexas de aplicação, técnicas de arquitetura para Big Data e, crucialmente, as melhores práticas de segurança ao utilizar lógica embutida no banco. O domínio destes conceitos separa desenvolvedores juniores de arquitetos de dados.

💡 Objetivo: Implementar análises temporais e ranking sem `GROUP BY`, organizar tabelas massivas e mitigar riscos críticos de Triggers/Procedures em ambientes corporativos.

1. Window Functions (Funções de Janela)

Permitem realizar cálculos em um conjunto de linhas relacionadas à linha atual, sem agregá-las em um único resultado. Usam a cláusula OVER().

-- Ranking de Vendas por Vendedor -- ROW_NUMBER: Número sequencial único por partição -- RANK: Mesma posição para empates, pula o próximo SELECT vendedor_id, mes_venda, valor_vendido, ROW_NUMBER() OVER(PARTITION BY vendedor_id ORDER BY valor_vendido DESC) AS row_num, RANK() OVER(PARTITION BY vendedor_id ORDER BY valor_vendido DESC) AS posicao FROM Vendas WHERE ano = 2023;

Ideal para paginar resultados ou identificar o "top 1" por categoria.

-- Análise Temporal: Comparação Mês a Mês SELECT mes, receita, LAG(receita, 1) OVER(ORDER BY mes) AS mes_anterior, ROUND((receita - LAG(receita, 1) OVER(ORDER BY mes)) / LAG(receita, 1) OVER(ORDER BY mes) * 100, 2) AS variacao_pct FROM Receita_Mensal;

LAG acessa a linha anterior e LEAD a próxima. Elimina a necessidade de `SELF JOIN`.

2. Common Table Expressions (CTEs)

A cláusula WITH cria resultados temporários nomeados que existem apenas durante a execução da query. Melhora drasticamente a legibilidade e manutenção.

-- CTE para pré-agregação antes do JOIN principal WITH Vendas_Agregadas AS ( SELECT cliente_id, SUM(valor) AS total_gasto, COUNT(pedido_id) AS qtd_pedidos FROM Pedidos WHERE data_criacao >= '2023-01-01' GROUP BY cliente_id ) SELECT c.nome, c.email, va.total_gasto, va.qtd_pedidos FROM Clientes c JOIN Vendas_Agregadas va ON c.cliente_id = va.cliente_id WHERE va.total_gasto > 5000 ORDER BY va.total_gasto DESC;
✅ Vantagem: O otimizador do banco pode materializar a CTE em memória temporária, reutilizando-a múltiplas vezes na mesma query sem recalcular.

3. Particionamento de Tabelas (Big Data)

Divide tabelas fisicamente grandes em partes menores (partições) com base em intervalos, listas ou hash. O motor do banco lê apenas a partição relevante, acelerando queries em milhões de linhas.

-- Partição por Ano (RANGE) CREATE TABLE Logs_Acesso ( id_log BIGINT, usuario VARCHAR(50), data_hora DATETIME, ip_origem VARCHAR(45) ) PARTITION BY RANGE(YEAR(data_hora)) ( PARTITION p2021 VALUES LESS THAN (2022), PARTITION p2022 VALUES LESS THAN (2023), PARTITION p2023 VALUES LESS THAN (2024), PARTITION p2024 VALUES LESS THAN (2025), PARTITION p_futuro VALUES LESS THAN MAXVALUE );

Consultas filtradas por `WHERE YEAR(data_hora) = 2023` escaneiam apenas 1/5 dos dados.

-- Partição por HASH (Distribuição uniforme) CREATE TABLE Sessoes_Usuario ( id_sessao CHAR(36) PRIMARY KEY, usuario_id INT, token VARCHAR(255), criado_em TIMESTAMP ) PARTITION BY HASH(usuario_id) PARTITIONS 8;

Ideal para sharding interno quando não há um critério temporal claro.

4. 🛡️ Segurança & Riscos: Triggers e Procedures

Crítico

O uso de lógica embutida no banco exige cautela extrema em ambientes de produção. Abaixo, os principais riscos e mitigações:

RiscoDescriçãoMitigação
Lógica OcultaRegras de negócio ficam invisíveis ao código da aplicação.Documentação rigorosa e versionamento no repositório (flyway/liquibase).
Dificuldade de DebugErros em cascata são difíceis de rastrear.Use `SIGNAL` com mensagens claras e logs em tabela de auditoria.
SQL InjectionProcedures com SQL dinâmico (`PREPARE/EXECUTE`) são vulneráveis.Use parâmetros tipados e evite concatenação de strings em queries dinâmicas.
Escalada de PrivilégiosTriggers podem operar com permissões do `DEFINER`.Cuidado com `SQL SECURITY DEFINER`. Prefira `INVOKER` onde possível.
PerformanceTriggers rodam transacionalmente, travando a tabela.Evite loops pesados. Use filas assíncronas (RabbitMQ/Kafka) para logs pesados.
Backup/RestoreScripts manuais podem perder a ordem de criação.Use ferramentas automatizadas. Nunca confie em `mysqldump` manual sem validação.
⚠️ Regra de Ouro: Mantenha a lógica de negócio na camada de aplicação sempre que possível. Use Procedures/Triggers apenas para integridade transacional, auditoria imutável ou otimizações extremas de I/O.
-- Exemplo Seguro: Procedure com Parâmetros Tipados e Auditoria DELIMITER // CREATE PROCEDURE sp_transferencia_segura( IN p_origem INT, IN p_destino INT, IN p_valor DECIMAL(10,2), OUT p_sucesso BOOLEAN ) BEGIN DECLARE v_saldo DECIMAL(10,2); -- Bloco transacional seguro DECLARE EXIT HANDLER FOR SQLEXCEPTION BEGIN ROLLBACK; SET p_sucesso = FALSE; INSERT INTO Audit_Log (acao, status, erro) VALUES ('TRANSFERENCIA', 'FALHA', SQLSTATE); END; START TRANSACTION; SELECT saldo INTO v_saldo FROM Contas WHERE id = p_origem FOR UPDATE; IF v_saldo >= p_valor THEN UPDATE Contas SET saldo = saldo - p_valor WHERE id = p_origem; UPDATE Contas SET saldo = saldo + p_valor WHERE id = p_destino; INSERT INTO Transacoes (conta_origem, conta_destino, valor, data) VALUES (p_origem, p_destino, p_valor, NOW()); SET p_sucesso = TRUE; ELSE SET p_sucesso = FALSE; END IF; COMMIT; END // DELIMITER ;
