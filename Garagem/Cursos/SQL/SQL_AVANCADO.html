SQL Avançado

Stored Procedures, Funções, Triggers e Otimização de Queries

Introdução

Avançado Prático

Nesta etapa, transcendemos as consultas básicas e manipulamos a lógica do banco de dados diretamente. Aprenderemos a encapsular regras de negócio em Stored Procedures e Funções, automatizar ações com Triggers transacionais, e utilizar ferramentas de análise de performance como EXPLAIN para transformar queries lentas em execuções otimizadas.

💡 Objetivo: Escrever SQL modular, seguro e performático, aplicando padrões de engenharia de dados em ambientes MySQL/PostgreSQL.

1. Subconsultas Correlacionadas & Máscaras

Subconsultas correlacionadas são executadas linha a linha pela consulta externa, permitindo filtragem dinâmica. Combinadas com funções de string, garantem conformidade com a LGPD.

-- Retorna nomes de pessoas que moram em bairros específicos -- A subconsulta é executada primeiro e os IDs são passados para o WHERE IN SELECT nome_pessoa FROM Pessoa WHERE idpessoa IN ( SELECT idpessoa FROM Endereco WHERE LOWER(bairro) = 'tropical ville' );
⚠️ Performance: IN com subconsultas grandes pode ser lento. Considere usar JOIN ou EXISTS para otimização em datasets massivos.
-- Máscara de CPF para conformidade com LGPD -- CONCAT une strings, SUBSTRING extrai partes específicas SELECT idpessoa, nome_pessoa, CONCAT(SUBSTRING(cpf, 1, 3), '****', SUBSTRING(cpf, 8, 11)) AS cpf_mascarado FROM Pessoa;

Essa técnica permite exportar relatórios sem expor dados sensíveis em texto plano.

2. Triggers Transacionais & Auditoria

Triggers garantem consistência automática. Em nível avançado, usamos OLD. e NEW. para acessar estados antes/depois da modificação, e blocos BEGIN...END para transações atômicas.

-- TRIGGER DE BACKUP: Salva registro antes da exclusão DELIMITER // CREATE TRIGGER backup_pessoa BEFORE DELETE ON Pessoa FOR EACH ROW BEGIN INSERT INTO Pessoa_backup (idpessoa, nome_pessoa, cpf, nascimento) VALUES (OLD.idpessoa, OLD.nome_pessoa, OLD.cpf, OLD.nascimento); END; // DELIMITER ;
🔍 Por que usar DELIMITER? ▼

Em stored procedures e triggers, usamos ; internamente. O DELIMITER // instrui o cliente SQL a tratar // como fim do comando, evitando execução prematura do bloco.

3. Stored Procedures (Procedimentos Armazenados)

Stored Procedures encapsulam lógica de negócio no servidor. Aceitam parâmetros IN, OUT e INOUT, reduzem tráfego de rede e centralizam regras de validação.

-- Procedimento para cadastro seguro com validação DELIMITER $$ CREATE PROCEDURE sp_cadastrar_pessoa( IN p_nome VARCHAR(255), IN p_cpf CHAR(11), IN p_nasc DATE, OUT p_status VARCHAR(50) ) BEGIN -- Declaração de variável local DECLARE v_existe INT; -- Verifica se CPF já existe SELECT COUNT(*) INTO v_existe FROM Pessoa WHERE cpf = p_cpf; IF v_existe > 0 THEN SET p_status = 'ERRO: CPF já cadastrado'; ELSE START TRANSACTION; INSERT INTO Pessoa (nome_pessoa, cpf, nascimento) VALUES (p_nome, p_cpf, p_nasc); COMMIT; SET p_status = 'SUCESSO: Pessoa cadastrada'; END IF; END$$ DELIMITER ; -- Execução SET @retorno = ''; CALL sp_cadastrar_pessoa('Maria Silva', '11122233344', '1990-05-10', @retorno); SELECT @retorno;
✅ Vantagem: O START TRANSACTION garante que a inserção só ocorre se a conexão estiver estável, evitando dados órfãos em caso de falha parcial.

4. Funções Definidas pelo Usuário (UDFs)

Diferente das Procedures, funções RETURNS um valor único e podem ser usadas diretamente em consultas SELECT, WHERE ou UPDATE.

-- Função que retorna idade completa baseada na data de nascimento DELIMITER $$ CREATE FUNCTION fn_idade_completa(p_nascimento DATE) RETURNS INT DETERMINISTIC BEGIN DECLARE v_idade INT; SET v_idade = TIMESTAMPDIFF(YEAR, p_nascimento, CURDATE()); RETURN v_idade; END$$ DELIMITER ; -- Uso direto em SELECT SELECT nome_pessoa, fn_idade_completa(nascimento) AS idade FROM Pessoa;
-- Função para formatar nome com iniciais maiúsculas DELIMITER $$ CREATE FUNCTION fn_formatar_nome(p_nome VARCHAR(255)) RETURNS VARCHAR(255) DETERMINISTIC BEGIN RETURN CONCAT(UCASE(LEFT(p_nome, 1)), LCASE(SUBSTRING(p_nome, 2))); END$$ DELIMITER ; UPDATE Pessoa SET nome_pessoa = fn_formatar_nome(nome_pessoa) WHERE idpessoa = 1;

5. Otimização de Queries com EXPLAIN

O comando EXPLAIN mostra o plano de execução do MySQL. Entender as colunas type, rows e key é essencial para transformar scans lentos em buscas indexadas.

Coluna EXPLAINO que IndicaIdeal
typeTipo de junção/buscaref, eq_ref, const
keyÍndice utilizadoNome do índice criado (não NULL)
rowsLinhas examinadasQuanto menor, melhor (próximo de 1)
ExtraDetalhes adicionaisEvitar Using filesort ou Using temporary
-- Analisando plano de execução EXPLAIN SELECT nome_pessoa, email FROM Pessoa P JOIN Contatos C ON P.idpessoa = C.idpessoa WHERE P.bairro = 'Tropical ville'; -- CRIANDO ÍNDICE PARA OTIMIZAR (Evita Full Table Scan) CREATE INDEX idx_bairro_pessoa ON Pessoa(bairro); CREATE INDEX idx_email_contato ON Contatos(email); -- Re-executando EXPLAIN após criar índices -- O type mudará de ALL para REF e rows cairá drasticamente EXPLAIN SELECT nome_pessoa, email FROM Pessoa P JOIN Contatos C ON P.idpessoa = C.idpessoa WHERE P.bairro = 'Tropical ville';
⚠️ Cuidado com Índices: Índices aceleram SELECT mas desaceleram INSERT/UPDATE/DELETE. Crie apenas nas colunas frequentemente usadas em WHERE, JOIN e ORDER BY.

🎯 Desafio Prático Avançado

Avaliativo

Desenvolva um script SQL completo para um módulo de `RH_Empresa` que inclua:

  • Crie uma Stored Procedure `sp_aumentar_salario(id_funcionario, percentual)` que atualize o salário, valide se o aumento é maior que 50%, e retorne status via parâmetro OUT.
  • Crie uma Função `fn_nome_completo(nome, sobrenome)` que retorne a string formatada.
  • Crie um Trigger `tr_log_alteracao` que registre na tabela `Audit_Salarios` o ID, valor antigo e novo sempre que um `UPDATE` ocorrer na tabela de salários.
  • Execute EXPLAIN em uma query que busca funcionários por departamento e sugira um índice para otimizá-la.
✅ Critérios: Uso correto de DELIMITER, OLD/NEW, transações, parâmetros de saída e análise de performance.
