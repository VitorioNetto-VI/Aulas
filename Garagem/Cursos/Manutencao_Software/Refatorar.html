1 Abertura e Conceitos Fundamentais

Por que testar ao refatorar? Pirâmide de testes e ciclo Red-Green-Refactor

A refatoração sem testes é como reformar uma casa sem alicerces: qualquer mudança pode causar danos invisíveis. Testes automatizados fornecem a rede de segurança necessária para modificar código com confiança.

Pirâmide de Testes

A pirâmide de testes define a proporção ideal entre diferentes tipos de testes:

Testes Unitários
Testes de Integração
Testes E2E
🏗️
Base da Pirâmide (70%)

Testes isolados de funções e métodos. Rápidos, baratos e determinísticos. Devem cobrir a maior parte do código.

Meio da Pirâmide (20%)

Testam a interação entre módulos, classes e serviços externos mockados. Garantem que as peças se encaixam corretamente.

🌐
Topo da Pirâmide (10%)

Testes de ponta a ponta que simulam o comportamento real do usuário. Mais lentos e caros, mas essenciais para cenários críticos.

Ciclo Red-Green-Refactor

Fase Ação Estado do Teste Objetivo
Red Escrever teste falhando ❌ Falha Definir comportamento esperado
Green Implementar código mínimo ✅ Passa Fazer o teste passar
Refactor Melhorar código mantendo testes ✅ Passa Limpar código sem mudar comportamento

Fixture Básica no conftest.py

Fixtures são funções que fornecem dados de teste reutilizáveis:

python 
# tests/conftest.py
import pytest
from snake import Snake

@pytest.fixture
def snake_inicial():
    """Retorna uma instância de Snake na posição inicial."""
    return Snake(posicao_inicial=[(5, 5)])

@pytest.fixture
def snake_com_cauda(snake_inicial):
    """Snake com 3 segmentos para testes de crescimento."""
    snake_inicial.crescer()
    snake_inicial.crescer()
    return snake_inicial
3 Mapeando uma Classe Existente

Testes de caracterização: descobrindo comportamento sem documentação

Quando enfrentamos código legado sem testes, o primeiro passo é criar testes de caracterização. Eles capturam o comportamento atual do sistema, permitindo que identifiquemos bugs e comportamentos inesperados antes de começar a refatorar.

️
Atenção

Testes de caracterização documentam o comportamento atual, não necessariamente o comportamento correto. Se um teste falhar durante a refatoração, verifique se o bug estava no código original ou na sua alteração.

Exemplo: Mapeando a classe Snake

Vamos descobrir como a classe Snake se comporta na prática:

python 
# tests/test_snake_characterization.py
import pytest
from snake import Snake

class TestSnakeCharacterization:
    """Testes de caracterização para entender o comportamento atual."""

    def test_posicao_inicial(self):
        snake = Snake(posicao_inicial=[(5, 5)])
        assert snake.get_cabeca() == (5, 5)
        assert snake.get_corpo() == [(5, 5)]

    def test_movimento_cima(self):
        snake = Snake(posicao_inicial=[(5, 5)])
        snake.mover('UP')
        assert snake.get_cabeca() == (5, 4)

    def test_crescimento_apos_comer(self):
        snake = Snake(posicao_inicial=[(5, 5)])
        tamanho_antes = snake.get_tamanho()
        snake.comer()  # Simula comer uma maçã
        tamanho_depois = snake.get_tamanho()
        assert tamanho_depois == tamanho_antes + 1

    @pytest.mark.xfail
    def test_colisao_com_parede(self):
        """Comportamento descoberto: lança exceção ao colidir."""
        snake = Snake(posicao_inicial=[(0, 0)])
        snake.mover('UP')  # Deveria colidir com parede superior
        # Se passar, o comportamento é diferente do esperado!
💡 Dica: Use pytest.mark.xfail

O decorator @pytest.mark.xfail marca testes que esperamos que falhem. Isso é útil para documentar comportamentos bugados que você ainda não corrigiu, sem quebrar o build.

4 Refatoração Guiada com TDD

Extrair método, tratar exceções, parametrizar e usar mocks

Agora que temos testes de caracterização, podemos começar a refatorar com segurança. Vamos aplicar técnicas de TDD para melhorar a estrutura do código.

1. Extrair Método

Identifique blocos de código complexos e extraia-os em métodos com nomes descritivos:

Antes (Código Original)
Depois (Refatorado)

def mover(self, direcao):
    # Código espaguete: muita lógica misturada
    x, y = self.posicoes[0]
    if direcao == 'UP':
        nova_pos = (x, y - 1)
    elif direcao == 'DOWN':
        nova_pos = (x, y + 1)
    elif direcao == 'LEFT':
        nova_pos = (x - 1, y)
    elif direcao == 'RIGHT':
        nova_pos = (x + 1, y)
    
    # Verificação de colisão misturada com movimento
    if nova_pos[0] < 0 or nova_pos[0] >= self.largura:
        raise ValueError("Colisão horizontal!")
    if nova_pos[1] < 0 or nova_pos[1] >= self.altura:
        raise ValueError("Colisão vertical!")
    
    self.posicoes.insert(0, nova_pos)
    if not self.crescendo:
        self.posicoes.pop()
python 
def mover(self, direcao):
    nova_cabeca = self._calcular_proxima_posicao(direcao)
    self._verificar_colisao(nova_cabeca)
    self._atualizar_corpo(nova_cabeca)

def _calcular_proxima_posicao(self, direcao):
    """Calcula a próxima posição da cabeça baseada na direção."""
    deltas = {
        'UP': (0, -1), 'DOWN': (0, 1),
        'LEFT': (-1, 0), 'RIGHT': (1, 0)
    }
    dx, dy = deltas[direcao]
    x, y = self.get_cabeca()
    return (x + dx, y + dy)

def _verificar_colisao(self, posicao):
    """Verifica se a posição está dentro dos limites do tabuleiro."""
    if not (0 <= posicao[0] < self.largura):
        raise ColisaoError("Colisão com parede horizontal")
    if not (0 <= posicao[1] < self.altura):
        raise ColisaoError("Colisão com parede vertical")

def _atualizar_corpo(self, nova_cabeca):
    """Atualiza o corpo da cobra com a nova posição."""
    self.posicoes.insert(0, nova_cabeca)
    if not self.crescendo:
        self.posicoes.pop()
    self.crescendo = False

2. Testes Parametrizados

Use @pytest.mark.parametrize para testar múltiplos cenários com um único teste:

python 
@pytest.mark.parametrize("direcao, delta_esperado", [
    ('UP', (0, -1)),
    ('DOWN', (0, 1)),
    ('LEFT', (-1, 0)),
    ('RIGHT', (1, 0)),
])
def test_calculo_posicao(self, direcao, delta_esperado, mocker):
    # Testa se o cálculo de posição está correto para todas as direções
    snake = Snake(largura=10, altura=10)
    mocker.patch.object(snake, 'get_cabeca', return_value=(5, 5))
    resultado = snake._calcular_proxima_posicao(direcao)
    esperado = (5 + delta_esperado[0], 5 + delta_esperado[1])
    assert resultado == esperado

5 Cobertura e Integração Contínua

pytest-cov, relatórios de cobertura e configuração do GitHub Actions

A cobertura de testes mede qual porcentagem do seu código é exercida pelos testes. Não é uma métrica perfeita, mas ajuda a identificar áreas não testadas.

Rodando com Cobertura

bash 
# Roda testes e gera relatório HTML
pytest --cov=src --cov-report=html

# Gera relatório no terminal com falha se cobertura < 80%
pytest --cov=src --cov-fail-under=80

Interpretando o Relatório

Métrica Significado Meta Recomendada
Stmts Total de linhas de código -
Miss Linhas não executadas 0
Cover Porcentagem de cobertura ≥ 80%
Branch Cobertura de ramificações (if/else) ≥ 75%
📊
Cuidado com a Obsessão por Cobertura

100% de cobertura não significa 100% de qualidade. Testes mal escritos podem cobrir código sem testar comportamento real. Foque em cobertura de cenários, não apenas de linhas.

GitHub Actions para CI

Configure um workflow para rodar testes automaticamente em cada push:

yaml 
name: Python Tests

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v3
      
      - name: Setup Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.10'
      
      - name: Install Dependencies
        run: |
          pip install -r requirements.txt
          pip install pytest pytest-cov
      
      - name: Run Tests with Coverage
        run: pytest --cov=src --cov-fail-under=80

Desafio: Refatorar Snake Game

Refatore o repositório garageunicorn/Refatorar-Snake aplicando todas as técnicas aprendidas.

Critérios de Avaliação

  • Separar classes em arquivos individuais (snake.py, food.py, game.py)
  • Cobertura de testes ≥ 80% para todas as classes
  • Uso de fixtures no conftest.py para reutilização
  • Testes parametrizados para cenários múltiplos
  • Mocks isolando dependências externas (input, random, etc.)
  • Código seguindo PEP 8 com type hints
  • Workflow GitHub Actions configurado e passando
💡 Dicas para o Desafio
  • Comece criando testes de caracterização antes de modificar qualquer código
  • Use pytest --lf para rodar apenas os testes que falharam na última execução
  • Documente decisões de refatoração em comentários ou README
  • Submeta um Pull Request com descrição detalhada das mudanças
